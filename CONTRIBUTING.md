# Contributing to VulnFlow

Thank you for your interest in contributing to VulnFlow.

VulnFlow is a visual smart contract audit pipeline builder. Contributions can include code, documentation, pattern files, workflow ideas, bug reports, and feedback from real audit use.

## Star the Project

If VulnFlow is useful to you, starring the repository is one of the easiest ways to support the project.

## Questions

If you have a question about usage, configuration, or workflow design:

- check the project documentation first
- open an issue if the answer is not documented
- contact us on X / Twitter: [@RightNowIn](https://x.com/RightNowIn)

## Share Ideas

Ideas for new features and improvements are welcome.

Examples:

- better audit workflow UX
- new pipeline blocks or behaviors
- better support for local models
- improvements to memory, docs search, or reporting
- better integrations with external research tools
- new pattern-checking workflows

When sharing an idea, explain the problem it solves and how it would improve real audit work.

## Report Bugs

Found a bug? Please open an issue.

Helpful bug reports usually include:

- a short description of the problem
- steps to reproduce it
- what you expected to happen
- what actually happened
- screenshots, logs, or error messages if relevant

If the issue is related to models, tools, or configuration, do not include real API keys or private secrets.

## Improve Documentation

Documentation improvements are always valuable.

Useful contribution areas:

- making guides easier for new users
- clarifying how Agent, Memory, Patterns, or `audit_docs` work
- fixing outdated examples
- improving explanations around `conf.yaml`
- improving screenshots or usage examples

## Contribute Pattern Files

You can also contribute YAML-based pattern files or improve existing ones.

Good pattern contributions are:

- clear
- reusable
- focused on a specific class of issues
- practical for real audit workflows

## Contribute Code

Code contributions are welcome across the project.

Examples:

- backend improvements
- frontend improvements
- pipeline logic improvements
- better error handling
- external tool integration improvements
- memory and docs workflow improvements
- performance and usability fixes

Please keep changes focused and easy to review.

## Before Opening a Pull Request

Before submitting a pull request:

1. Make sure the change is clear and limited in scope.
2. Update documentation if the behavior changed.
3. Test the affected part of the project locally.
4. Make sure you are not committing secrets or local-only configuration.

For local setup and project usage, see [`readme.md`](readme.md).

## Pull Request Guidelines

A good pull request should include:

- a clear title
- a short explanation of the change
- why the change is needed
- screenshots if the UI changed
- testing notes if relevant

Small and focused pull requests are preferred over large mixed changes.

## Security Notes

Please do not commit:

- API keys
- private `conf.yaml` values
- personal credentials
- local-only secrets or environment details

## Be Respectful

Please keep communication constructive, clear, and respectful.

## Thank You

Every contribution helps, whether it is a bug report, a documentation fix, a pattern update, or a code improvement.
